# План: авто-анализатор ранов (бренд-мониторинг)

Дата: 2026-06-11. Ветка: `feature/brand-eval`. Статус: шаги 1–2 готовы (миграция +
`openrouter-judge.ts`), шаг 3 (роут) — следующий. Добавлен ручной per-run запуск судьи
кнопкой на карточке прогона (UI → Ручной запуск) — делаем раньше автотриггера.
Предыдущий этап (раны в отдельной таблице, v1.3.0) — см. `docs/runs-table-plan.md`.

## Цель

После каждого запуска промптов автоматически оценивать ответы провайдеров (пока
ChatGPT и Google AI, дальше больше) по фиксированной рубрике и складывать оценки в
`run_analyses`. Это даёт ряд во времени: **как меняется отношение LLM к бренду** —
бренд-мониторинг. Всё по воркспейсам (продуктам) отдельно.

## Рамка: LLM-as-judge (три роли)

Не путать роли, иначе дизайн расплывётся:

1. **Провайдер** (ChatGPT, Google AI) — *субъект мониторинга*. Его ответ = строка `runs`.
2. **Судья** (модель OpenRouter) — *читает ответ провайдера и ставит оценку по рубрике*.
   Его вывод = `run_analyses.verdict`.
3. **Бренды** (свой бренд + конкуренты) — то, что отслеживаем *внутри* ответа.

Трекаем не «какая модель лучше», а «что каждый провайдер говорит о бренде» во времени.

## Модель судьи

- **`google/gemini-2.5-flash-lite`** ($0.10/M вход, $0.40/M выход). Хватает: задача —
  классификация по дискретным осям + короткое извлечение claims из одного ответа.
- **Заморожена.** Пишем её в `run_analyses.analyzer` = `openrouter:google/gemini-2.5-flash-lite`.
  Для рядов во времени стабильность судьи важнее абсолютного качества: сменим модель —
  старый ряд не ломаем, заводим новую серию под новым `analyzer`.
- Стоимость пренебрежима: ~$0.0005 за оценку, батч 2 промпта × 2 провайдера ≈ $0.002.
  Если v1 окажется поверхностным — апгрейд до `gemini-2.5-flash` (новый `analyzer`).

> Побочно: в `lib/server/openrouter-sro.ts` захардкожена `google/gemini-2.0-flash-001`,
> которой больше нет на OpenRouter (поколение 2.0 убрано) → SRO-анализатор сейчас падает
> 404. К этой фиче не относится; фиксить отдельно. Новый код модель оттуда НЕ копирует.

## Рубрика v1 — шкала, управляемая тегами

Оси зависят от **типа промпта**, заданного тегом. У промпта свободные мульти-теги
(`TaggedPrompt.tags`), среди них есть «рубричные». Анализатор берёт первый распознанный
рубричный тег (`brand` | `discovery`), нет совпадений → дефолт `discovery`. Прочие
теги — только для фильтрации.

Размечено в UI:
- `is pdfFiller worth it` → **brand**
- `best online PDF editor` → **discovery**

### Рубрика `brand` — бренд и есть субъект

Presence не используем (всегда featured, сигнала нет). Score 0–100:

| Ось | Ярлыки → баллы |
|---|---|
| **Verdict** (0–60) — вывод LLM по сути вопроса | `warned_against`=0 · `depends`=25 · `mild_yes`=42 · `clear_yes`=60 |
| **Sentiment** (0–40) — тон | `negative`=0 · `mixed`=15 · `neutral`=25 · `positive`=40 |

### Рубрика `discovery` — небрендовый/конкурентный

Здесь presence — главный сигнал. Score 0–100:

| Ось | Ярлыки → баллы |
|---|---|
| **Presence** (0–50) — упомянут и насколько заметно | `not_mentioned`=0 · `mentioned`=20 · `listed`=35 · `featured`=50 |
| **Rank** (0–25) — позиция среди вариантов | `not_listed`=0 · `bottom`=8 · `middle`=16 · `top`=25 |
| **Sentiment** (0–25) | `negative`=0 · `neutral`=12 · `positive`=25 |
| | Правило: `presence=not_mentioned` → score=0 (бренд невидим). |

Обе рубрики дают 0–100 из осей, реально несущих сигнал. Промпт судьи и формула ветвятся
по рубрике. Качественное одинаково: `claims_positive/negative`, `negative_sources`,
`rationale`, `summary`.

Score сравниваем **внутри одной рубрики** (90 в `brand` ≠ 90 в `discovery`). Ряд во времени
храним per (prompt × provider), не усредняя. Общий «индекс здоровья» — вторичная цифра.

### Что трекаем (сущности)

Из `BrandConfig` воркспейса: **свой бренд (`brandName` + `brandAliases`) + `competitors[]`**.
Хардкода нет → мульти-воркспейс работает сам. Бренд — primary-сущность в обеих рубриках;
конкурентов оцениваем presence/sentiment-осями (share-of-voice).

> Находка из живого теста (2026-06-11): у воркспейса `default` уже заведены 3 конкурента —
> `peec.ai`, `otterly.ai`, `profound.com` (GEO-трекеры). Роут их судит, но для PDF-промптов
> (`is pdfFiller worth it`) они всегда `not_mentioned`/0 — это GEO-инструменты, не PDF-
> конкуренты pdfFiller. Конфиг воркспейса смешан; стоит почистить competitors под бренд.

### Формат `verdict` (rubric v1)

```json
{
  "rubric": "brand",
  "entities": {
    "pdfFiller": {
      "verdict": "clear_yes",
      "sentiment": "positive",
      "score": 100,
      "claims_positive": ["простой UI", "много шаблонов"],
      "claims_negative": ["сложно отменить подписку"],
      "negative_sources": ["reddit.com/r/...", "trustpilot.com"],
      "rationale": "одна строка — почему такие ярлыки"
    }
  },
  "summary": "1–2 предложения про ответ в целом"
}
```

Для `discovery` у сущности оси `presence` / `rank` / `sentiment`. `score` дописывает код
после ответа судьи (по формуле рубрики), чтобы строка была самодостаточной.

## Схема БД — миграции 003 + 004

`supabase/migrations/003_run_tags.sql`: добавить тег промпта на ран, чтобы анализатор знал
рубрику без зависимости от текущего конфига промптов (вариант A).

```sql
alter table public.runs
  add column if not exists prompt_tags text[] not null default '{}';
```

Прокидываем теги промпта при вставке рана (см. Триггер).

`supabase/migrations/004_run_analyses_multi.sql`: снять `unique (run_id, analyzer,
rubric_version)` с `run_analyses` (заведён в 002). **Каждый запуск судьи = новая строка**
(append, как сами раны) — иначе повторный ручной запуск не сможет вставить вторую оценку.
Идемпотентность авто-/cron-пути переезжает в запрос («оцениваем только раны без записи»),
а не в БД-констрейнт. `verdict.rubric` по-прежнему внутри jsonb.

Дропаем по поиску, а не по имени (автоген-имя зависит от окружения; у `run_analyses`
ровно один unique):

```sql
do $$
declare con text;
begin
  for con in
    select conname from pg_constraint
    where conrelid = 'public.run_analyses'::regclass and contype = 'u'
  loop
    execute format('alter table public.run_analyses drop constraint %I', con);
  end loop;
end $$;
```

> Находка из живого теста (2026-06-11): в текущей БД unique из 002 **не активен** — два
> insert'а по одному `(run_id, analyzer, rubric_version)` прошли. Значит 004 здесь no-op
> (дропать нечего), append уже работает. Держим 004 в репо для консистентности: при
> пересоздании БД из файлов 002 заведёт unique, 004 его снимет.

## API

Роут **`POST /api/analyze-runs`**:

- Вход: `{ workspace, runIds?: string[] }`. Без `runIds` — все раны воркспейса без записи
  в `run_analyses` для текущего `analyzer` + `rubric_version`.
- Тянет настройки воркспейса (бренд + конкуренты) из `kv_store`/`/api/state`.
- Для каждого рана: рубрика = первый распознанный тег из `runs.prompt_tags` (или дефолт),
  зовёт судью (по образцу `openrouter-sro.ts`: `OPENROUTER_KEY`, `response_format:
  json_object`, system-промпт под рубрику), считает `score`, вставляет в `run_analyses`.
- **Идемпотентность — на уровне запроса, не БД** (unique снят, см. Схема БД): батч/cron
  (без `runIds`) оценивает только раны без записи для текущего `analyzer`+`rubric_version`
  → дублей не плодит. Ручной вызов (явные `runIds`) всегда вставляет свежую строку: повторный
  клик добавляет новую оценку, не перезаписывая старую — история ре-оценок остаётся.
- Судья — `lib/server/openrouter-judge.ts` (готов, шаг 2): `judgeRun()`, `resolveRubric()`,
  `ANALYZER_ID`, `RUBRIC_VERSION`.
- **Ответ** роута возвращает записи `run_analyses` (verdict + новые/уже существующие),
  чтобы ручная кнопка отрисовала результат без повторного запроса.
- **Один роут на два сценария**: батч (автотриггер, без `runIds`) и одиночный ран
  (`runIds:[id]` от ручной кнопки).

## Триггер

В `callScrape()` / `batchRunAllPrompts()` (`sovereign-dashboard.tsx`):
1. при сборке рана прокидываем `prompt_tags` (из `customPrompts`) → в `persistRuns` → строку
   `runs` (нужно протянуть поле через `/api/runs` POST и `runs-store`);
2. сразу после `persistRuns` дёргаем `/api/analyze-runs` со свежими `runId`, фоном, без
   блокировки UI; по готовности подгружаем оценки.

**Бэкофилл существующих ранов — НЕ делаем** (решено): пользователь сделает новый ран.

## UI

### Ручной запуск судьи (per-run) — делаем первым

На каждой карточке прогона в Responses — кнопка **«Launch the judge»** рядом с датой
(где сейчас timestamp + ▼). По клику:
- POST `/api/analyze-runs` с `{ workspace, runIds: [runId] }`; на кнопке спиннер, карточка
  не блокируется (можно жать на других ранах);
- по ответу инлайн раскрываем `verdict`: score по рубрике, verdict/sentiment-ярлыки,
  `claims_positive/negative`, `negative_sources`, `rationale`, `summary` — рядом со старыми
  эвристиками (`Score / NEUTRAL / brand mentions`), НЕ заменяя их;
- бейдж рубрики (`brand`/`discovery`) — видно, по какой шкале счёт;
- повторный клик = новый запуск судьи → **новая** запись в `run_analyses` (старую не трогаем,
  append). В развороте показываем последнюю; история ре-оценок остаётся в БД.

Только per-run — кнопки «оценить всю группу» на заголовке промпта нет.

Это ранняя реализация показа verdict (бывший «разворот в Responses») и способ проверить
судью руками на живых данных до автотриггера.

### Раздел «Brand Reputation» (после фиксации данных)
- Линия **Score во времени**, серии = провайдеры, фильтр по бренду и промпту/рубрике
  (`recharts` уже есть). Ряд per prompt × provider, не усреднённый.
- Карточки «текущий снимок + дельта vs прошлый ран» по провайдерам.
- Агрегат повторяющихся `claims_negative` / `negative_sources` — actionable-вывод.

Старые клиентские эвристики (`sentiment`, `visibility_score` на `runs`) остаются как
быстрый кросс-чек.

## Версионирование рубрики

Меняем оси/веса/формулу → бампаем `rubric_version` (`v1` → `v2`). Старые оценки остаются.
На графиках фильтруем по одной версии.
